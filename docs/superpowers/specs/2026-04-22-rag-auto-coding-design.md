# RAG 自动编码优化设计

## 背景

当前自动生成编码流程由 `EnhancedCodingGenerator` 负责组织：先对句子生成一阶编码，再通过 `SemanticMatcher` 将一阶编码匹配到编码库中的二阶编码，并进一步匹配三阶编码。`CodingLibraryManager` 负责读取 `coding_library.json`，`GroundedTheoryCoder` 负责构建最终三级编码结构并自动编号。

现有方案的问题不只是检索精度，而是编码基数扩散：如果每条句子级一阶编码都直接进入二阶和三阶匹配，即使引入 RAG，也可能把大量语义相近的一阶编码分散挂载到不同二阶或三阶，导致上层编码数量过多。

本设计采用“聚类优先的本地 RAG 编码器”：先压缩一阶候选基数，再对代表性一阶主题执行本地混合检索和高置信匹配，最后通过固定兜底类控制低置信结果。

## 目标

- 提升一阶编码抽象质量，使一阶编码更稳定、更少重复。
- 控制二阶和三阶编码数量，避免低置信匹配造成编码体系膨胀。
- 使用本地 RAG，不把分析文本、编码库或派生索引发送到外部服务。
- 保留 `coding_library.json` 作为业务主库，不在第一版中自动改写、删除、合并编码库内容。
- 以低侵入方式扩展现有 `SemanticMatcher` 和 `EnhancedCodingGenerator` 流程。
- 保证自动编码完成后，用户仍可点击主界面的“手动编码”进入 `ManualCodingDialog`，对自动生成的三级编码进行增删改操作。
- 保证自动编码与手动编码的现有联动行为不变，包括自动编码缓存、`self.structured_codes` 传递、手动编辑后替换或合并自动编码结果。
- 在编码库被编辑后，自动更新或失效派生 RAG 索引，确保后续自动编码使用最新编码库信息。
- 按硬件能力选择运行策略，使独显、高性能 CPU、集成显卡和轻薄本都能稳定运行。

## 非目标

- 第一版不自动新增、删除、重命名二阶或三阶编码。
- 第一版不引入外部向量数据库服务。
- 第一版不把分析文件原文写入长期向量库或持久化 RAG 文档。
- 第一版不替换 `GroundedTheoryCoder` 的编号与结构构建职责。
- 第一版不改变主界面进入手动编码的交互路径，不移除当前手动编码对自动编码结果的替换和合并能力。
- 第一版不强制依赖 GPU；GPU 只能作为可用时的加速路径，CPU 必须始终可用。

## 编码库 RAG 适配性判断

`coding_library.json` 可以作为 RAG 的原始知识源，但不建议原样直接向量化使用。

本地统计结果显示：

- 编码库字段结构完整，三阶编码包含 `id`、`name`、`description`、`second_level_codes`，二阶编码包含 `id`、`name`、`description`、`third_level`、`third_level_id`。
- 当前规模约 2000 个三阶编码、2975 个二阶编码，文件约 1.1 MB，适合本地轻量索引。
- 三阶编码下二阶数量中位数为 1，说明部分三阶编码粒度偏细。
- 三阶描述中位长度约 13 字，二阶描述中位长度约 10 字，直接向量化时语义信息偏少。
- 二阶名称存在较多重复或弱区分表达，直接检索容易产生混淆。

因此第一版应创建派生 RAG 索引，而不是直接把原始编码条目作为唯一检索文档。

## 推荐架构

新增一个派生索引层，保留原编码库不变：

```text
coding_library.json
  -> RagIndexBuilder
  -> cache/rag_index/code_documents.json
  -> cache/rag_index/token_index.json
  -> cache/rag_index/vector_embeddings.npz
  -> cache/rag_index/index_meta.json

EnhancedCodingGenerator
  -> 一阶候选生成
  -> FirstLevelClusterer
  -> RAGSemanticMatcher
  -> 二阶高置信匹配
  -> 三阶映射回挂
  -> GroundedTheoryCoder
  -> structured_codes
  -> ManualCodingDialog
```

### 派生 RAG 文档

每个二阶编码和三阶编码都会被转换为更适合检索的语义文档。二阶文档应包含：

- 编码层级：二阶
- 编码 ID、名称、定义
- 所属三阶 ID、名称、定义
- 同组三阶下的兄弟二阶名称
- 从名称和描述中提取的关键词
- 规范化检索文本

示例：

```text
层级：二阶编码
编码ID：1.1
编码名称：自我效能与成就感
定义：员工希望通过创新证明自己的能力，获得心理满足感或职业自豪感
所属三阶：个体动机与心理驱动
三阶定义：描述员工为什么要进行越轨创新的内在原因
同组三阶下相关二阶：任务紧急性倒逼、隐性资源利用
检索关键词：自我效能、成就感、能力证明、心理满足、职业自豪
```

这样可以弥补原始描述过短的问题，提高向量检索稳定性。

## 数据流

### 1. 一阶候选生成

继续复用当前 `abstract_sentence` 流程，包括规则抽取和可选的 `abstract_reranker_latest` 重排模型。此阶段输出仍是句子级一阶候选，但它们只是候选，不立即进入最终结构。

### 2. 一阶聚类压缩

新增一阶聚类步骤，把表达不同但语义相近的一阶候选合并为主题簇。聚类特征来自：

- 一阶候选文本的向量相似度
- token 关键词重叠
- 原句局部上下文
- 候选文本规范化结果

每个簇生成一个代表性一阶编码。代表性文本优先选择语义完整、长度适中、支持句子数较多的候选；必要时使用 RAG 检索到的二阶定义辅助润色，但不引入外部生成服务。

### 3. 二阶混合检索

对每个一阶簇代表文本执行混合检索：

- 先用 token 倒排索引召回候选二阶，减少向量计算范围。
- 再用本地向量相似度排序。
- 最后计算综合分，综合分由向量相似度、token 重叠、簇支持度、编码库权重组成。

只有高置信候选才能挂载到具体二阶编码。

### 4. 三阶映射回挂

三阶不再对所有三阶编码自由语义匹配。默认通过二阶编码的 `third_level_id` 或 `third_level` 回挂到三阶。

只有在二阶映射缺失、映射冲突或编码库字段异常时，才启用三阶语义匹配作为兜底判断。

### 5. 低置信固定兜底

低支持度、低相似度、语义冲突的候选不直接生成新的二阶或三阶编码。

固定兜底规则如下：

- 二阶匹配失败或不稳定时，进入 `其他各类话题`。
- 三阶映射失败或冲突时，进入 `其他重要维度`。
- 低支持度簇即使存在弱相似候选，也不扩展编码体系，优先进入固定兜底类。

进入 `其他各类话题` 的条件：

- 二阶混合检索综合分低于阈值。
- 二阶 top1 与 top2 分差过小，语义边界不清。
- 一阶聚类支持句子数太少，且没有强关键词命中。
- token 检索和向量检索指向不同二阶，存在语义冲突。

进入 `其他重要维度` 的条件：

- 二阶编码没有可靠 `third_level_id` 或 `third_level` 映射。
- 同一个二阶候选映射到多个三阶且无法判定。
- 三阶语义兜底匹配低于阈值。
- 三阶 top1 与 top2 分差过小。

### 6. 自动编码与手动编码联动

RAG 优化只能改变自动生成编码的内部生成与匹配策略，不能改变自动编码结果进入手动编码界面的方式。

必须保持以下行为：

- 自动编码完成后，主界面仍保存 `self.structured_codes`。
- 自动编码完成后，仍保存带一阶编码标记的自动编码缓存文本。
- 用户点击“手动编码”时，`ManualCodingDialog` 仍接收当前文件内容、自动编码结构和自动编码缓存。
- 手动编码界面必须能对自动生成的三阶、二阶、一阶编码进行新增、删除、重命名、移动和内容调整。
- 手动编码完成后，仍保留当前的“完全替换自动编码”与“合并手动编码和自动编码”能力。
- RAG 聚类产生的簇信息可以作为内部元数据保存，但不能破坏 `GroundedTheoryCoder` 和 `ManualCodingDialog` 已依赖的三级编码结构。

输出结构兼容要求：

- `一阶编码` 仍以当前调用方可识别的 key 映射保存。
- `二阶编码` 仍保存二阶名称到一阶 key 列表的映射。
- `三阶编码` 仍保存三阶名称到二阶名称列表的映射。
- `file_sentence_mapping` 仍原样返回，保证手动编码和跨文件导航继续可用。

### 7. 编码库编辑后的索引更新

编码库编辑包括新增、删除、修改二阶编码，新增、删除、修改三阶编码，以及从备份恢复编码库。

每次 `CodingLibraryManager` 成功保存编码库后，派生 RAG 索引必须自动失效或更新。第一版优先采用安全的“失效标记 + 下次使用前懒重建”策略，而不是每次编辑后立即阻塞式全量重建。

索引更新策略：

- `CodingLibraryManager.save_library()` 成功后写入索引失效标记，或调用轻量的 `RagIndexManager.invalidate()`.
- 下次自动编码开始时，检查 `index_meta.json` 中的编码库哈希、更新时间和索引版本。
- 若索引失效，优先在后台或进度提示阶段重建派生索引。
- 若重建失败，回退到现有语义匹配或 token 检索，不能阻塞用户继续编码。
- 若只新增或修改少量编码，后续版本可以做增量更新；第一版允许全量重建，因为当前编码库规模较小。

索引更新不能修改 `coding_library.json` 的业务内容。派生索引目录 `cache/rag_index` 可删除、可重建、可视为缓存。

### 8. 硬件自适应运行策略

系统最终面向不同用户电脑，运行策略必须根据硬件能力自动降级。

硬件分层：

- 高性能策略：检测到可用 NVIDIA 独立显卡和 CUDA 时，向量编码、批量相似度计算和可选重排模型优先使用 GPU。
- 标准策略：没有 CUDA 但 CPU 性能和内存充足时，使用 CPU 批量处理，适当降低 batch size。
- 轻量策略：集成显卡、轻薄本、低内存或电池供电环境下，优先使用 token 召回、较小 batch size、较小候选集和更保守的聚类规模。
- 保底策略：向量模型不可用或加载失败时，使用 token 检索、关键词匹配和现有规则回退流程。

运行策略选择必须自动完成，并允许通过配置覆盖。策略选择不能要求用户理解 GPU、CUDA 或模型细节。

性能约束：

- 自动编码不应一次性把所有句子和所有编码都放入 GPU 或内存。
- 向量计算必须分批执行。
- 轻量策略下应减少 `RAG_TOKEN_TOP_K`、`RAG_VECTOR_TOP_K`、batch size 和最大聚类候选数。
- 索引重建必须显示进度或日志，避免界面看起来卡死。
- 无论使用 GPU 还是 CPU，输出结果结构必须一致。

## 主要组件

### RagIndexBuilder

负责从 `CodingLibraryManager` 加载的编码库生成派生 RAG 索引。

职责：

- 构建二阶和三阶检索文档。
- 提取关键词并构建 token 倒排索引。
- 生成或增量更新向量矩阵。
- 写入索引元数据，包括编码库路径、文件更新时间、哈希、模型名、索引版本。

索引失效条件：

- `coding_library.json` 文件哈希变化。
- sentence-transformer 模型路径或名称变化。
- RAG 索引版本变化。

### RagIndexManager

负责管理派生索引生命周期。

职责：

- 检查索引是否存在、是否过期、是否与当前编码库哈希一致。
- 在编码库编辑成功后标记索引失效。
- 在自动编码开始前按需触发索引重建。
- 在索引重建失败时提供明确错误原因，并允许生成流程回退。
- 清理临时索引文件，避免半写入索引被后续流程误用。

### RAGSemanticMatcher

继承或组合现有 `SemanticMatcher`，保持原匹配接口尽量兼容。

职责：

- 加载派生 RAG 索引。
- 执行 token 召回、向量排序、综合打分。
- 返回候选编码、相似度、token 命中、冲突原因、是否满足高置信门控。
- 当索引不可用时，回退到现有 `SemanticMatcher` 行为。

### FirstLevelClusterer

负责对一阶候选进行去重和语义聚类。

职责：

- 合并完全重复或规范化后重复的一阶候选。
- 合并向量相似度高且关键词重叠明显的候选。
- 为每个簇保留来源句子、文件信息、支持句子数。
- 输出代表性一阶编码和簇内明细。

### CodingDecisionPolicy

集中管理阈值和兜底策略，避免门控逻辑散落在生成器里。

职责：

- 判断二阶候选是否高置信。
- 判断三阶映射是否可靠。
- 判断是否进入 `其他各类话题` 或 `其他重要维度`。
- 输出可解释的决策原因，便于日志和后续调参。

### RuntimeStrategyDetector

负责检测硬件能力并选择运行策略。

职责：

- 检测 CUDA 是否可用、GPU 名称、可用显存、CPU 核心数和内存规模。
- 生成运行配置，例如设备、batch size、候选集大小、是否启用重排模型、是否启用向量聚类。
- 当 GPU 运行失败或显存不足时自动切换到 CPU 策略。
- 在日志中记录策略选择原因，方便排查不同电脑上的性能差异。

## 隐私与空间策略

- 派生 RAG 索引只基于编码库构建，不持久化分析文件原文。
- 分析文件文本只在运行时参与一阶生成、聚类和检索。
- 临时运行缓存随生成器生命周期释放，或仅保存不可逆的短期向量缓存。
- 索引目录位于 `cache/rag_index`，可安全删除并自动重建。
- 第一版使用 `json + npz + token_index`，不引入外部数据库进程。
- 编码库编辑后只更新派生索引或索引失效标记，不持久化分析文件内容。
- 轻量策略下限制缓存和 batch size，避免轻薄本长时间高负载或内存占用过高。

## 错误处理与回退

- RAG 索引构建失败时，记录日志并回退到现有语义匹配流程。
- 向量模型不可用时，使用 token 检索和现有关键词回退逻辑。
- 编码库字段异常时，不自动修复主库，异常候选进入固定兜底类。
- 聚类失败时，保留原一阶候选并继续执行保守匹配。
- GPU 初始化失败、CUDA 不可用、显存不足或模型加载失败时，自动切换到 CPU 或轻量策略。
- 派生索引重建失败时，不影响用户进入手动编码界面继续编辑已有自动编码结果。

## 测试策略

### 单元测试

- RAG 文档构造包含二阶、三阶、父子关系和兄弟二阶信息。
- 索引元数据在编码库哈希变化后触发重建。
- token 召回能命中关键词相关的二阶候选。
- 一阶聚类能合并重复和高相似候选。
- 二阶低分、低 margin、低支持度、检索冲突时进入 `其他各类话题`。
- 三阶映射缺失或冲突时进入 `其他重要维度`。
- 编码库保存成功后，索引元数据失效或哈希变化能被检测到。
- 运行策略检测能在 CUDA 可用、CUDA 不可用、低内存三类情况下给出不同配置。

### 集成测试

- 使用小型测试编码库和多条相似句子，验证多个句子先聚成少量一阶主题。
- 验证高置信一阶簇挂载到预期二阶，并通过 `third_level_id` 回挂三阶。
- 验证低置信输入不会生成新的二阶或三阶。
- 验证 RAG 关闭时现有流程仍可运行。
- 验证自动编码完成后，主界面仍能打开 `ManualCodingDialog` 并加载自动编码结果。
- 验证手动编码完成后，仍可选择完全替换自动编码或合并编码结果。
- 验证编码库编辑后，下次自动编码会重建或刷新派生索引。

### 回归测试

- 保证返回结构仍包含当前调用方依赖的键：`一阶编码`、`二阶编码`、`三阶编码`、`file_sentence_mapping`。
- 保证 `GroundedTheoryCoder` 仍能对结果编号。
- 保证 `CodingLibraryManager` 的增删改查行为不受派生索引影响。
- 保证 `ManualCodingDialog` 对自动编码结构的增删改入口不受 RAG 聚类元数据影响。
- 保证 CPU-only 环境下自动编码可完成，只是速度较慢。
- 保证 GPU 环境和 CPU 环境输出结构一致。

## 验收标准

- 在相似句子重复较多的输入中，最终一阶编码数量少于句子级候选数量。
- 低置信二阶统一进入 `其他各类话题`，不产生新的二阶名称。
- 低置信三阶统一进入 `其他重要维度`，不产生新的三阶名称。
- 高置信二阶优先使用编码库已有二阶名称。
- 三阶优先由二阶 `third_level_id` 或 `third_level` 决定。
- 删除 `cache/rag_index` 后系统能自动重建索引。
- 全流程不依赖外部服务。
- 自动编码完成后点击“手动编码”，能进入手动编码界面对自动生成结果进行增删改。
- 手动编辑后的结果仍能替换或合并自动编码结果。
- 编码库编辑并保存后，派生索引不会继续使用旧编码库内容。
- 独显 CUDA 环境优先使用 GPU 加速；无独显、集成显卡或轻薄本环境自动使用 CPU 或轻量策略。
- GPU 不可用或运行失败时，系统自动降级，不中断自动编码流程。

## 设计自审

- 无需改写原编码库，降低数据损坏风险。
- 派生索引只包含编码库内容，不持久化分析原文，满足本地隐私要求。
- 先聚类再匹配，直接解决一阶基数导致二三阶扩散的问题。
- 二阶和三阶都有固定兜底类，避免低置信结果创造新上层编码。
- 三阶优先映射回挂，避免三阶自由匹配造成语义漂移。
- 第一版使用轻量本地索引，符合当前编码库规模，不引入过重依赖。
- 自动编码输出结构保持兼容，手动编码入口和增删改能力不被 RAG 流程改变。
- 编码库编辑后通过索引失效和懒重建保证派生索引一致性，同时避免每次编辑都阻塞界面。
- 硬件策略自动降级，保证不同用户电脑都能运行，只在可用时使用 GPU 加速。
